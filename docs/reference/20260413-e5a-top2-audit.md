# E5A Top2 替换审计方案

## 1. 实验定位

E5A v1 不是新的训练实验, 也不是只读已有结果表的复盘.

它的定位是:

- 第一阶段不训练新模型.
- 基于已有 checkpoint 新跑一遍带额外记录的 audit pass.
- 目标是判断: 当前 `top2` code 选择是否已经是主要瓶颈.

当前主线固定为:

- `dense-t025`
- `top2 read`
- `den_aware selector`
- `shared_den`

E5A v1 只审这条主线, 不扩展到其他 remote formula / merge mode / selector mode.

## 2. 当前主线与层范围

E5A v1 必须把 layer scope 写死.

在当前 Zoology 主线配置里:

- 模型由 `n_layers=2` 个 `TransformerBlock` 组成.
- `sequence_mixer` 使用 `Hybrid([conv, flash_vqg])`.
- `Hybrid` 按 `layer_idx % len(configs)` 选择 mixer.

因此当前主线里实际只有一个 `FlashVQGMixer`, 且它位于最后一个 `TransformerBlock` 的 `sequence_mixer` 位置.

E5A v1 默认采用单层审计:

- `audit_layer_idx = 1`, 即当前主线中唯一的 `FlashVQGMixer` 所在层.
- layer-local 指标只针对该层计算.
- `pred_mode` 表示仅在该层替换 Top2 set 后, 继续完成真实 forward 得到的最终预测.

这里的 `audit_layer_idx = 1` 是当前主线定义的一部分, 不是 E5A v1 需要 sweep 的自由参数.

若后续要研究全层同步替换或逐层 sweep, 应单独立项, 不混入 E5A v1 定义.

## 3. E5A 要回答什么

E5A 要回答的问题不是“某个新 heuristic 看起来好不好”, 而是:

> 如果只替换当前 remote read 的 Top2 code set, 而不改变当前 remote 内部近似与 local 路径, 最终输出和 MQAR 结果会不会明显更好?

如果答案是“会”, 则当前瓶颈更像 Top2 code selection.  
如果答案是“不会”, 则当前瓶颈更像 code 内近似或 remote 内部估计.

## 4. 明确不做什么

E5A v1 不做以下事情:

- 不训练新的 experiment 5 模型.
- 不修改当前训练主路径语义.
- 不引入新的 kernel 或 Triton 审计实现.
- 不把 `query` / `ref` 接进训练图.
- 不只靠已有 `history.csv` / `summary.json` / `metrics_index.json` 反推结论.
- 不把所有 token 都混进评估.
- 不把“单层审计”和“全层同步替换”混成一个定义.

E5A v1 只做:

- 读取已有 checkpoint.
- 新跑一次带额外中间量导出的 eval / audit pass.
- 构造五种 Top2 集合.
- 替换 remote 侧 Top2 集合后, 重算 layer-local audit 输出.
- 在真实 forward 中接回后续计算, 得到最终 `pred_mode`.
- 输出一组足够硬的聚合指标表.

## 5. 当前机制的正确表述

不要把当前机制写成:

> “先选 top2, 再在 top2 内单独归一化, 得到 remote 输出, 然后再和 local 合并.”

更准确的表述是:

1. 先为当前 row 选出 Top2 code.
2. remote 分支只保留这两个 code 的贡献.
3. local 分支保持不变.
4. remote 与 local 一起进入最终 shared denominator 合并.

还必须注意一点:

- remote 与 local 在 phase2 中共用同一个数值稳定化最大值 `m_shared`.
- 这个 `m_shared` 不是只在 Top2 内重新计算的局部 max, 而是当前 row 上 remote 与 local 共同使用的公共 max.
- E5A v1 若做 runtime override, 只能替换 Top2 code set, 必须沿用同一个 `m_shared` 机制, 不能改成“只在被选中的 Top2 内重新做一遍 softmax”.
- `mode` 只负责决定“选哪些 code”, 不直接手工指定新的 read 权重.
- 一旦 Top2 code set 被替换, 被选中的 code 会继续按当前主线 forward 的既有规则自然得到读出权重, 未选中的 code 权重则变为 0.

在当前 `clr_v1 + shared_den` 主线里, 更贴实现的底层量是:

- `Num_far`
- `Den_far`
- `Num_loc`
- `Den_loc`

最终输出是:

$$
h = \frac{Num_{far} + Num_{loc}}{Den_{far} + Den_{loc}}
$$

其中 remote 权重的稳定化应沿用:

$$
m_{shared} = \max(m_{remote}, m_{local})
$$

也就是说, remote 路径的权重计算应保持为 `\exp(logit - m_{shared})`, 而不是在被选中的 Top2 内重新定义一个新的归一化基准.

若把 mode 对应的 Top2 集合写为 $\mathrm{Top2}_{mode}(r)$, 则更贴实现的有效读出权重可以理解为:

$$
w^{eff}_{mode}(r,s) = w_{base}(r,s) \cdot \mathbf{1}[s \in \mathrm{Top2}_{mode}(r)]
$$

这里的含义是:

- `w_base` 仍按当前主线的既有读出规则计算.
- `mode` 改变的是保留哪些 code.
- E5A 不在新 Top2 子集内额外重新定义一套权重公式.

因此 E5A 中的 `o_remote_audit` 不是模型长期保存的新状态, 而是 audit-only 派生量:

$$
o^{remote}_{audit} = \frac{Num^{far}_{audit}}{Den^{far}_{audit} + \varepsilon}
$$

## 6. 五种 Top2 集合

E5A v1 固定审以下五种 Top2 集合:

- `student`
- `write`
- `time`
- `query`
- `ref`

其中:

- `student`: 当前模型真实 forward 中实际使用的 Top2.
- `write`: 只看 write-side 质量分布.
- `time`: 在 write-side 基础上再考虑时间项.
- `query`: 看 query-key 匹配驱动下哪些 code 更该被选.
- `ref`: 在 query-key 匹配和时间项同时成立时, 哪些 code 更该被选.

`ref` 是 E5A 中最接近 oracle 的 audit-only 集合, 但不是训练目标.

## 7. row 集合与 remote-visible scope

MQAR 中只有真正 query 位置的标签不是 `-100`, 其余位置都应忽略. 定义:

$$
\text{task\_rows} = \{ r \mid y_r \neq -100 \}
$$

其中:

- 任务 accuracy
- `pred_changed_vs_student`

都只在 `task_rows` 上统计.

机制审计再进一步收紧为:

$$
\text{audit\_rows} = \text{task\_rows} \cap \{ r \mid J_r^{remote} \neq \varnothing \}
$$

其中:

- Top2 overlap
- `ref_mass_capture`
- `top1_top2_margin`
- `remote_dist_to_ref`
- `final_dist_to_ref`

都只在 `audit_rows` 上统计.

设输入按 block 组织为 `[B, H, N, L, ...]`, 其中 $T = N \cdot L$.  
当前主线 remote path 的因果边界定义为:

$$
idx_{\text{remote}}(n) = \max(0, n - (w - 1))
$$

其中 $w = local\_num\_blocks$.

因此 block $n$ 内任一 row 的 remote-visible history 终点为:

$$
hist\_end(n) = idx_{\text{remote}}(n) \cdot L
$$

对应的 remote-visible causal history 是:

$$
J_r^{remote} = \{ j \mid j < hist\_end(n) \}
$$

`write / time / query / ref` 四种 score 都必须在同一个 $J_r^{remote}$ 上定义.

## 8. 统一的 code score 定义

后文统一使用 $z^{mode}_{r,s}$ 表示 row $r$ 对 code $s$ 的审计分数, 其中 $mode \in \{write, time, query, ref\}$.

### 8.1 Write

$$
z^{write}_{r,s}
=
\log \sum_{j \in J_r^{remote}} \pi_{j,s}
$$

### 8.2 Time

$$
z^{time}_{r,s}
=
\log \sum_{j \in J_r^{remote}} \pi_{j,s} \exp(c_r - c_j)
$$

### 8.3 Query

$$
z^{query}_{r,s}
=
\log \sum_{j \in J_r^{remote}} \pi_{j,s} \exp(q_r^\top k_j)
$$

### 8.4 Reference

$$
z^{ref}_{r,s}
=
\log \sum_{j \in J_r^{remote}} \pi_{j,s} \exp(q_r^\top k_j + c_r - c_j)
$$

其中:

- $\pi_{j,s}$ 对应 weighted routing write 的 `weights[..., s]`.
- $q_r$ 是当前 row 的 query 向量.
- $k_j$ 是历史 token $j$ 的 key 向量.
- $c_r, c_j$ 是 CLR path 的时间项.

因此要按上式精确计算 `z_write / z_time / z_query / z_ref`, 除了当前文档已列出的 `Q_blk`, `c_q`, `c_end`, `hist_end` 之外, 还必须拿到:

- write-side routing 权重, 即 `\pi_{j,s}` 对应的 `weights`.
- 历史 token 级 key 张量, 即 `K_blk` 或等价物.
- 历史 token 级时间张量, 即 `c_all` 或等价物.

## 9. 从层内审计到最终预测

E5A 必须区分两类输出.

### 9.1 层内审计输出

对任意候选 Top2 集合, 定义:

$$
o^{remote}_{audit} = \frac{Num^{far}_{audit}}{Den^{far}_{audit} + \varepsilon}
$$

$$
h^{layer}_{audit} = \frac{Num^{far}_{audit} + Num^{loc}}{Den^{far}_{audit} + Den^{loc} + \varepsilon}
$$

其中:

- `o_remote_audit` 只用于 remote-level 误差比较.
- `h_audit_layer` 只用于 layer-local audit.

### 9.2 模型级最终输出

`h_audit_layer` 不能直接等同于最终任务预测.  
`pred_mode` 的来源必须是:

1. 在真实 model forward 中进入 `audit_layer_idx`.
2. 用 mode 对应的 `top2_codes` 替换当前 student Top2.
3. 用替换后的 remote contribution 形成新的该层输出 `h_audit_layer`.
4. 将该层输出继续送入模型剩余未改动的计算.
5. 最终经 lm head 得到 `logits_mode`.
6. 定义:

$$
pred^{mode} = \arg\max logits^{mode}
$$

7. 再在 `task_rows` 上计算 accuracy.

必须明确:

> 替换 Top2 set 的动作发生在真实 forward 内, 然后继续完成后续计算得到最终预测; `h_audit_layer` 只用于机制指标, 不直接等同于任务预测.

## 10. 所需中间量与责任边界

E5A 需要大量逐行, 逐头, 逐 code 的中间张量.  
现有 scalar metrics 链路不足以承载这些张量:

- `FlashVQGMixer.get_scalar_metrics()` 只保留标量.
- `Trainer` 只汇总标量 metrics.

因此 E5A v1 需要额外提供 audit-only tensor capture helper.

### 10.1 `Flash-VQG` 负责导出的原始张量

必需导出:

- `Q_blk`
- `K_blk`, 或等价的 token 级 history key 张量
- `write_weights`, 即 RoutingVQ `weights` 对应的 $\pi_{j,s}$
- `c_all`, 或等价的 token 级时间张量
- `c_q`
- `c_end`
- `student_top2`
- `student_top1_top2_margin`
- `Num_loc`
- `Den_loc`
- `local_max`
- `m_shared`, 即当前 row 上 remote / local 共用的稳定化 max
- `g_remote`
- `R_remote`
- `L_remote`
- `a_remote`
- `codebook`
- `clr_basis`
- `hist_end`

可选导出:

- `student_top2_logits`

不要求导出:

- 全 code 的 `student_logits`

### 10.2 `zoology` 负责构造和计算的量

由 `zoology` 侧负责:

- `task_rows`
- `audit_rows`
- `z_write`
- `z_time`
- `z_query`
- `z_ref`
- `top2_write`
- `top2_time`
- `top2_query`
- `top2_ref`
- 聚合指标表与 `summary.json`

其中:

- `z_write / z_time / z_query / z_ref` 在 `zoology` 中基于 `write_weights`, `K_blk`, `c_all`, `Q_blk` 和 `hist_end` 计算.
- `c_q / c_end` 继续保留, 主要用于和当前 remote path 的边界与时间偏置保持一致.
- layer-local 的 `o_remote_audit / h_audit_layer` 若在 `zoology` 侧重算, 必须复用同一个 `m_shared`, 不能对 override 后的 Top2 子集单独重做局部 softmax.

### 10.3 责任边界

最合适的边界是:

- `Flash-VQG`: 导出张量与 runtime override 所需 helper.
- `zoology`: 基于这些张量完成 E5A 指标计算, 汇总和实验输出.

## 11. override 机制与实现约束

E5A v1 不要求引入训练主路径的永久配置项.

更推荐的做法是:

- 在 Flash-VQG attention 层实现 audit-only runtime helper 或 hook.
- 当 audit pass 开启时, 指定层临时接收外部 `top2_codes` override.
- 默认训练与普通 eval 路径完全不受影响.

当前 Zoology 主干 forward 不沿整条模型链显式透传 audit 参数, 因此 E5A v1 更推荐采用模块级 runtime 上下文触发, 而不是把 audit 开关做成训练配置项或新增一整条 forward 参数链.

推荐的最小接口形态是:

- 在 `FlashVQGMixer` / `FlashVQGAttention` 上提供 `set_audit_runtime(runtime)` 与 `clear_audit_runtime()`.
- `runtime` 只承载本次 audit forward 的临时控制信息, 例如:
  - `enabled`
  - `capture_tensors`
  - `mode_name`
  - `override_top2_codes`
  - `capture_sink`

触发方式应为:

1. `e5a_audit.py` 在进入目标层 forward 前临时设置 `runtime`.
2. 跑一次正常 model forward.
3. forward 结束后立即清理 `runtime`.

`FlashVQGAttention` 在 forward 内对 `runtime` 的使用约束是:

- 若 `runtime is None` 或 `runtime.enabled == False`, 必须完全走现有 baseline 路径.
- 若 `runtime.capture_tensors == True`, 只额外导出审计所需中间张量.
- 若 `runtime.override_top2_codes` 非空, 只替换 remote 侧 Top2 保留集合.
- 不得借机改动 `m_shared`, read 权重公式, local path, shared denominator merge 或其他默认计算路径.

E5A v1 的实现约束是:

- 禁止 row-wise / code-wise Python loop.
- 优先采用张量化实现.
- 允许沿 block, history, batch 或 case 维做 chunked vectorization.

具体建议:

- `write / time`: 优先 prefix snapshot.
- `query / ref`: 优先 blockwise vectorization.
- 当全量 `[B,H,N,L,T]` 张量过大时, 允许沿 `T` 或 batch 维做 chunked vectorization.

这不改变数学定义, 只改变工程实现形态.


- `aggregate_by_case.csv`

`row_audit.parquet` 只用于追查异常行.

主表至少包含:

- `mode`
- `acc_total`
- `acc_512x128`
- `acc_1024x256`
- `overlap_vs_student`
- `overlap_vs_ref`
- `ref_mass_capture`
- `top1_top2_margin`
- `remote_dist_to_ref`
- `final_dist_to_ref`
- `pred_changed_vs_student`

默认核心 case:

- `512x128`
- `1024x256`

所有 `acc_*` 指标都必须与现有 Zoology `valid/accuracy` 保持同一聚合口径:

1. 对每个样本, 仅在该样本的 `task_rows` 上计算 token-level mean accuracy.
2. 再对样本做均值聚合.

也就是说, `acc_total`, `acc_512x128`, `acc_1024x256` 都采用 per-example mean 再 overall mean, 而不是把所有有效 row 直接混成一个 row-weighted mean.

## 13. 裁决规则

### 13.1 定义 student 明显偏离 ref

满足任一条即可:

- `overlap_vs_ref(student) <= 0.60`
- `ref_mass_capture(student) <= 0.75`

### 13.2 定义 ref 替换带来有意义提升

在核心 case `512x128` 和 `1024x256` 上同时满足:

- 平均 `acc_ref - acc_student >= 1.0pp`
- 任一单 case 不低于 `0.5pp`
- 不存在单 case 回退超过 `0.3pp`

### 13.3 定义 ref 替换提升有限

同时满足:

- 平均 `acc_ref - acc_student < 0.3pp`
- 两个核心 case 的增益都小于 `0.5pp`

### 13.4 定义 student 已经比较接近 ref

同时满足:

- `overlap_vs_ref(student) >= 0.80`
- `ref_mass_capture(student) >= 0.90`

### 13.5 最终裁决

判定 `selection_bottleneck`:

- 满足 13.1
- 且满足 13.2

判定 `in_code_approx_bottleneck`:

- 满足 13.4
- 且满足 13.3

其他情况统一归为:

- `mixed_or_inconclusive`

`remote_dist_to_ref` 与 `final_dist_to_ref` 作为佐证项使用, 不单独设置绝对裁决阈值.

## 16. 一句话执行摘要

E5A v1 是一个不训练的 Top2 替换审计实验.  
它固定当前主线 checkpoint, 在当前主线中唯一的 `FlashVQGMixer` 上构造 `student / write / time / query / ref` 五种 Top2 集合, 并在不改变 local 路径与 remote 近似公式的前提下, 只替换 remote 侧 Top2 code set, 重算 `o_remote_audit` 与 `h_audit_layer`, 再接回真实 forward 得到 `pred_mode`.  
通过 `overlap_vs_ref`, `ref_mass_capture`, `top1_top2_margin`, `remote_dist_to_ref`, `final_dist_to_ref` 与 MQAR accuracy 的联合变化, 判断当前瓶颈更像 Top2 code selection, 还是更像 code 内近似问题.
